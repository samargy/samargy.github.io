Technical Aspects

Below is the directory structure for my site. Having a good directory structure is good for rapid development, as time spent searching on where resources are, is time spent not developing. It consists of a public folder - containing all non .css or .html resources, e.g. images and fonts. After the public folder there is the 4 HTML files, each with their own respective content. Then you have websystems.css which contains the CSS for the entire site, this means all the styling across the site uses the same style sheet. This is great for both developmental purposes (everything is changed at once) and also design purposes (consistency).

Now that the file structure has been layed out. Its time to talk a bit about the page structure itself. Below is a table of all my main CSS classes & ids, and their respective purpose.

Name Type Purpose
h1 element Styles the title at top of webpage
body element Styles the background of the site
.body-container class Styles the grey/yellow box most of my sites content lies in
.main-heading class Styles the "Comments" piece of text you see on this page. Could be changed to an ID, although kept as class for possible reuse.
.sub-heading class Styles the sub headings you see around the site, e.g Technical Aspects, Aesthetics.
#navbar ul id + element First using an ID to specify the element. Then styling the element. This part of the navbar styled is the unordered list containing its li elements.
#navbar ul li a id + element First using an ID to specify the element. Then styling the element. This part of navbar styled is each list item e.g Home, Past, Future. This styling also has another version of it with :hover added on which controls what styling occurs when the users mouse hovers over the list item.

So what was the reasoning behind whether to style an element, a class or an id? Well, quite simply it came down to what the styling was needed for, and if it would be used again. If the styling was a one off for a big feature that would never be used anywhere else in the website, I chose to style the element specifically. If the styling needed to applied in a variety of places - then I made it a class. If it was a one off but could be used again then I made it an ID. (also it was in the specification to use an ID somewhere).

Key Technical Decisions I made:

As this is a Web Systems course and not an English course I thought I'd outline some of the other more specific technical choices I made in dot point format, rather than writing out an essay for each one.

  • Choosing to organise my files by using a public folder. - this is good for a scalable website, having the structure in place already means if your site wants to scale up and add more features then you have the structure ready.
  • Making the title style a font not an image - leads to faster load times
  • Stucture my website in a top-down column like format. This enables the website when viewed on mobile or skinner screens to condense and resize easier. This is expanded on more in Accessibility
  • Hosting the webpage using GitHub Pages, this enabled for rapid implementation and hosting of the site. GitHub Pages gives you a free hosting plan per account for static content - perfect for this site.

Finally, the advantage of having such a "raw" site is the peformance of it. Nowadays sites are far too bloated with tons of 3rd party add ons, this adds to the load times. Aesthetics

As stated earlier on this page, the main theme I was aiming for was a retro laid-back style. This allows me to make the site minamalistic, whilst also maintaining a sense of character.

To keep the site from feeling overcrowded as well as avoiding too much contrast (feels bad on the eyes), the colour palette was kept small - to 4 colours to be exact. These colours being:

  • ■Main Background - Dark Slate
  • ■Secondary Background - Deep Matte Grey
  • ■Main Text - Silver Fox
  • ■Secondary Text - Are ya yellow?!

So why these colours? Well other than the fact I have no deisgn skills, I wanted firstly to have a nice dark background, dark slate was a good choice for this. Following this I had an idea of a sort of "container box". This is the box you are reading in now, a good choice for the background of this "box" was the Deep Matte Grey, slightly lighter than the dark slate but still with a nice feel to it. Now onto the text, I played around with a couple colours for a bit, but nothing felt right. Then I went to white, and it looked better than the others but still wasn't to my liking. So I did the thing all web devs do best which is googling stuff - "what colour reads well on grey", the answer I got. Silver. Finally, I had to choose the secondary text colour. I went through a series of trials, red, green, blue, purple, pink, but everything seemed off. I finally arrived at yellow, and thought it fit the retro theme well.

Key Aesthetic Decisions I made:

  • Making the navbar elements change colour as they are hovered. - This provides a sense of reactivity for users.
  • Choosing to have a content box (the one you are reading in now). Many old blog sites had these so I thought it would fit the retro theme.
  • Choosing the font - "Gameplay" for the title. This gave it a sort of arcade vibe, aligning with the retro theme.
  • Styling the table with the border-collapse property so you get single lines instead of double lines. This aligns with the minamalistic approach.
  • Coming up with the "timeline" feature on the Past page. This delivered the content in a streamlined, simple and easy to read manner.
  • Making use of the scroll-behaviour CSS property. Setting this to smooth makes in page nav links jump to the section specified in a smooth manner. Example here.
  • Using the headings, sub headings and sub-sub-headings format. This allows for the infomation to be easily absorbed and not confusing on where to find the info.
  • For all text apart form the header and navbar, Times New Roman has been chosen. This is to invoke the feeling of a "raw" site, most earlier low traffic sites on the web had no font specified, and the default font for HTML was Times New Roman. Hence why I'm using it on this site now.

Accessibility

When it comes to accessibility its best to let other people think of the checklist that needs to be covered as you might miss or forget something on your own. To do this I used the W3C Accessibility Principles.

Screen Readers and Internal Structure

Due to screen readers not being able to read and view media, alt text is needed for non-text content. Therefore all images have a specified alt text. This also functions as a way for audio readers to read out the whole webpage, these images can be read aloud via their alternate text. Aside from just alt text, proper HTML structure has been enforced. This includes the use of standard HTML tags such as <h1> and <p> both of which have their own default formatting. Also, table heads <th> have also been used so that the tables maintain structure as well. This means when a stylesheet cannot be used, the page still maintains a "readable" format. Below is an example of this.

Mobile First Development

This site was designed with the Mobile-First Approach. The mobile first approach is useful in the fact developing screens for mobile is harder. Therefore, if done first it can make the rest of development easier. One of the more noticeable aspects of the mobile design is the single image per div of the site. This means you will never find two images side by side, one of the reasons is because of ease of development - a flexbox can just be used instead of multiple media queries (one for each major screen size). This also means that as you shrink the page horizontally (try it now) all the images change size slowly (due to their width properties being set using relative 'view height') and still remain centered. Due to the mobile first approach taken with the site, it actually reads better on mobile than on desktop in my opinion. Thats not saying it reads bad on desktop, just it reads even better on mobile. Fonts

Across the site, all fonts have been kept to the web standard size - this can be specified with a percentage. In doing so all text scales up or down accordingly when the zoom function is used on your browser. Zooming can be performed using Ctrl + and Ctrl - to zoom in and out respectively. This is great for users with poorer vision that need the text size just that tad bit bigger. On top of this, proper spacing has been put in place so text is not too close to the edges of the screen, as well as spacing between one another. Finally, styling tags have been used such as <b> for bolding text and <em> for italicizing them. This is good because if the stylesheet can't be accessed then some styling can still be applied - this is specifically for screen readers.

Colour Blindness and Colour Contrast

As well as aethetic purposes, the colours of the website have been chosen to meet colour contrast standards properly. The use of bright text on a dark background has more contrast than the vice versa. This is because your eyes absorb light, black is the absence of light, so when you are reading black text on a white background you are looking at the absence of light. Therefore, I chose the aforementioned bright text on dark. Aside from just basic readability purposes, these colour choices actually contribute to whether the site is viewable for colour blind users. There are three main types of colour blindness, absence of red cones, absence of blue cones, and absence of green cones. These are the cones in your eyes that detect a certain type of light. To test for these three different colour blind types, you can turn your site "grayscale", this takes mostly all colour out so you can just see the contrast. For this I used the Color Blind Web Page Filter Below is a side by side comparison.

As you can see the grayscaled site on the right is still quite readable. This means the contrast is high enough that even the most colour blind of people can still be able to use and read the website (albiet not to its maximum potential).

Usability

Finally, usability was considered carefully throughout the whole development proccess. One of the main features is when hovering over "clickable" elements the colour changes. This includes all anchor tags, as well as the navbar. In doing so this creates a more intuitive user experience. Also, another feature is that with external links, when clicked open a new page. This is a great usuability feature as the reader can go and check out the other page, and then close it and come back to the page they were just on. This also means that they are still scrolled down to where they were up to reading, as opposed to using the back feature.
